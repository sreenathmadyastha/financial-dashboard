Release Process — Pre-Release Checklist

All 6 gates must be completed before a production release is approved. Missing any gate = release is blocked. Exceptions require CISO sign-off with a formal waiver.

Details

01 Change Request (CR) ▼

Raise in your ITSM tool (ServiceNow / Jira SM) at least 5 business days before release.

  • Include: version, deployment window, impact assessment, rollback plan
  • Change type: Standard / Normal / Emergency
  • CAB approval required for Normal and Emergency changes
  • Post-deploy: update CR to Implemented and capture PIR
Done when: CR status = APPROVED in ITSM with CAB sign-off recorded.
02 Cyber Risk Management (CRM) Approval ▼

Security team reviews residual risk and validates controls are sufficient.

  • Submit: security questionnaire, threat model / DFDs, dependency inventory
  • Covers: auth controls, TLS, secrets management (Key Vault), tenant isolation, SIEM coverage
  • All Critical/High vulns must be remediated or formally risk-accepted with expiry date
Done when: Formal CRM approval email/ticket received from CISO office.
03 Archer Risk Request ▼

Raise an IT Risk record in the Archer GRC portal, linked to the CR.

  • Document identified risks with likelihood/impact ratings
  • Map to control library; assign risk treatment: Mitigate / Accept / Transfer
  • Attach evidence: scan reports, CRM approval, pen test certificate
  • Retain Archer Record ID for release package
Done when: Archer record = Approved/Closed. Record ID in release docs.
04 Penetration Test ▼

Conducted by internal security team or approved third-party on the staging environment.

  • Scope: all new/modified API endpoints, auth flows, tenant isolation boundaries
  • Methodology: OWASP Testing Guide v4 + OWASP API Security Top 10
  • Retest required for all High/Critical findings before sign-off
  • Deliverable: executive summary + detailed findings report + retest report
Done when: Final pen test report issued; all Critical/High findings closed or risk-accepted.
05 Fortify SAST Scan ▼

Runs via CI pipeline (GitHub Actions / Harness). Results uploaded to Fortify SSC.

  • Critical — must remediate before release. Blocks.
  • High — remediate or obtain CRM risk acceptance. Blocks unless accepted.
  • Medium / Low — track in backlog. Does not block.
  • False positive suppression requires AppSec team review
Done when: Zero Criticals; all Highs resolved or accepted; scan report in release package.
06 MAST / SAST / DAST Scans ▼

SAST (SonarQube / Checkmarx) — CI gate on every PR to release branch. Quality gate must pass; security hotspots triaged with AppSec.

DAST (OWASP ZAP / Burp Suite) — Against staging. Covers all authenticated + unauthenticated routes. Run within 5 business days of release date.

MAST (MobSF / Checkmarx MAST) — Required only if release includes mobile (iOS/Android). If no mobile component, mark N/A with justification in CR.

Done when: All applicable scans complete; no unaddressed Critical/High; reports attached; AppSec sign-off obtained.
